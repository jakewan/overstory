## [Unreleased]

### Added

- `project_summary`'s recommendation candidates and `backlog_review`'s deferred issues now each also carry a `blockedBy` field: the ascending, distinct numbers of the issue's still-open native GitHub blocked-by edges — the authoritative dependency signal, complementing the heuristic `bodyRefs` both already expose. Closed blockers are omitted (a closed blocker no longer gates), and a pull request can never appear (the edge is issue-to-issue), so unlike `bodyRefs` the open/closed state is read straight from the edge and needs no `openIssueSet` resolution. Cross-repository blockers are dropped — a foreign repo's issue number would otherwise collide with a local one. A companion `blockedByTruncated` flag marks an issue whose native edges exceeded the fetch window, where an empty `blockedBy` is not proof the issue is unblocked. Non-nil (serialized as `[]`) when an issue has no open native blockers. (#77)
- `project_summary`'s recommendation candidates and `backlog_review`'s deferred issues now each also carry a `blocking` field — the reverse direction of `blockedBy`: the ascending, distinct numbers of the issue's still-open downstream issues, the native GitHub edges this one is declared to block, so a caller can weigh how much downstream work an issue stands in front of, not just whether the issue is itself blocked. Same authoritative-edge semantics as `blockedBy`: closed downstream issues are omitted (a closed issue is no longer gated), a pull request can never appear (the edge is issue-to-issue), and cross-repository edges are dropped. A `blocking` edge is a gate this issue *contributes*, not necessarily the sole one — a downstream issue several issues block stays blocked until all of them close, so closing this issue is necessary but not always sufficient to free it. A companion `blockingTruncated` flag marks an issue whose native edges exceeded the fetch window. Non-nil (serialized as `[]`) when an issue gates nothing open. (#78)
- `project_summary`'s recommendation candidates and `backlog_review`'s deferred issues now each also carry a `subIssues` field plus a `subIssuesTotal`/`subIssuesCompleted` completion pair — the third native GitHub dependency relationship, parent → sub-issue hierarchy: a parent gated on its open children is not startable, so an age-driven "what's next" ranking can demote it instead of floating it as ready. `subIssues` is the ascending, distinct numbers of the issue's still-open child issues, with the same authoritative-edge semantics as `blockedBy`/`blocking` (completed children omitted, a pull request can never appear, cross-repository children dropped), non-nil (serialized as `[]`) when there are none, and a companion `subIssuesTruncated` flag when the native children exceeded the fetch window. The completion pair is GitHub's authoritative `subIssuesSummary`, counted over *all* children (every repository, never capped), so `subIssuesTotal − subIssuesCompleted` is an upper bound on the open children and may exceed the `subIssues` list — an empty `subIssues` with a positive gap signals a parent gated entirely by cross-repository or beyond-window children, not a ready issue. The fetch needs no preview request header (the sub-issue fields are generally available). (#80)
- `backlog_review` now also returns a `prTrajectory` reduction: for each of the same manifest-declared lookback windows the issue `trajectory` uses (in days), how many pull requests were opened and how many were closed within the last that-many days, and the net (opened minus closed) — the change-request closure-ratio signal for whether the project keeps pace with incoming PRs or accumulates an unreviewed PR backlog. Closed counts both merged and closed-without-merge PRs (a merged PR is closed); the block returns these counts, not a computed ratio, so the caller derives whatever ratio it presents. Windows are cumulative `last-N` lookbacks (a `last-7` PR also counts in `last-30` and `last-90`), reusing `trajectory.windows` (default `[7, 30, 90]`) and `trajectory.fetchLimit` (default `500`). Like the issue trajectory it reduces a dedicated second fetch — pull requests open and closed/merged by recent update — so a count's `fetchTruncated` flag marks when that fetch's cap was hit before the widest window was covered, making the counts lower bounds; if the fetch fails, the block degrades to `available: false` with an `unavailable` reason (e.g. `rate_limited`, `fetch_failed`) and the other blocks still return rather than failing the whole review. A reopened PR reads as open (GitHub clears its close), so a close later reopened does not count as outflow — the net-as-of-now reading. The block is aggregate, so the per-call `limit` does not apply to it. (#75)
- `project_summary` and `backlog_review` now each return an `openIssueSet` block: the ascending, distinct set of open issue `numbers` in the fetched window, with a `fetchTruncated` flag. It is the resolvable surface for the `bodyRefs` both tools already expose — a caller intersects a candidate's (or deferred issue's) stated `#N` dependencies against `numbers` to tell a ref naming a live open issue in this repo from one that does not, so an age-driven "what's next" ranking can demote a candidate gated behind an open sibling instead of floating it to the top. The set is same-repo, open, and issues-only, and is the full fetched window — never capped by `limit` — so a real open blocker beyond a block's list cap still resolves. The honesty bound is **presence gates; absence does not clear**: a ref in `numbers` names a live open issue, but a ref absent from it is not proof of resolution — it may be a closed issue, an open PR (PRs share the number space), a cross-repo reference, or, when `fetchTruncated`, an open issue beyond the window. (#71)
- `maintenance_activity` MCP tool: given an explicit `owner`/`repo`, an `author` login, and a window (`since`, with an optional `until` defaulting to now, both RFC3339), returns the state-mutation maintenance that user paid to existing issues and pull requests — the relabeling, milestoning/demilestoning, deferral-labeling, closing/reopening, assigning, and renaming that authored-activity counts structurally miss (a grooming afternoon produces near-zero authored counts). It returns the touched items, most-recently-touched first, each with `isPullRequest` (the stream mixes issues and PRs — the tool stays tag-blind and the caller splits the mix) and the actor's qualifying mutations in chronological order, each event carrying its `type`, instant, per-type payload (label name, milestone title, assignee login, or rename before/after), and a `viaAutomation` flag (set when GitHub attributes the event to an app — with the blind spot that an automation acting as the measured login is still attributed to it). The actor is matched by login string, so an unknown or inactive login yields zero items, never an error. A `truncated` flag marks a window the fetch could not fully cover. The `rateLimit` budget is the REST core pool (requests/hour) — a different pool from `authored_activity`'s GraphQL points, so the two are not comparable; a window ending far in the past over-reads from now and can report `truncated` with no in-window items. Like `authored_activity` it reads no manifest conventions and inherits the operator's `gh` credentials, so it can measure private repositories; it runs over a single `owner`/`repo` per call. (#69)
- `maintenance_activity_batch` MCP tool: the batched form of `maintenance_activity` — given a list of `owner`/`repo` slugs, one `author` login, and a window, it fans out the per-repo measure concurrently and returns one entry per repository in request order, each either the touched items (grouped as in the single-repo tool) or a per-repo unavailability marker (`not_found`, `rate_limited`, `fetch_failed`, `not_attempted`). Repositories are independent: a missing or failed one degrades to its own marker rather than failing the batch, and the result carries a single aggregated `rateLimit` — the tightest remaining across the successful repositories, or a throttle's reset instant when any repository was throttled (the REST core pool, not comparable with `authored_activity_batch`'s GraphQL points). Under load the fan-out adapts: a throttle trips batch-wide backpressure (the batch stops launching new fetches, so the not-yet-started repositories report `not_attempted` while in-flight fetches still complete), and each repository's fetch carries its own deadline so one hung repository degrades to `fetch_failed` without stalling the batch. Unlike `authored_activity_batch` there is no whole-batch author error — the actor is matched by login string, so an unknown login simply yields zero items per repository; an empty or oversized `repos` list, a malformed or duplicate slug, and an unparseable or inverted window are each rejected before any fetch. It reads no manifest conventions and inherits the operator's `gh` credentials, and returns per-repo facts only — merging across repositories and the attention verdict stay caller-side. (#69)
- `project_summary`'s recommendation candidates now each carry a `bodyRefs` field: the distinct `#N` references parsed from the issue's body, ascending, with pull-request references and the issue's own number excluded. These are the issue's stated dependencies, so a caller's "what to start next" ranking can tell a ready issue from one gated behind open siblings — keeping an age-driven heuristic from floating a terminal synthesis issue to the top on age alone. Parsed from GitHub's rendered plaintext body (a heuristic proxy for stated cross-references, not GitHub's native blocked-by/sub-issue edges), and non-nil (serialized as `[]`) when an issue references nothing. (#59)
- `authored_activity_batch` MCP tool: the batched form of `authored_activity` — given a list of `owner`/`repo` slugs, one `author` login, and a window, it fans out the per-repo measure concurrently and returns one entry per repository in request order, each either the six decomposed counts (with their per-category `fidelity` labels) or a per-repo unavailability marker (`not_found`, `rate_limited`, `fetch_failed`, `not_attempted`). Repositories are independent: a missing or failed one degrades to its own marker rather than failing the batch, and the result carries a single aggregated `rateLimit` — the tightest remaining across the successful repositories, or a throttle's reset instant when any repository was throttled. Under load the fan-out adapts: a throttle trips batch-wide backpressure — to avoid amplifying the rate limit the batch stops launching new fetches, so the not-yet-started repositories report `not_attempted` (in-flight fetches still complete) — and each repository's fetch carries its own deadline so one hung repository degrades to `fetch_failed` without stalling the batch. Because the author login resolves globally (independent of repository), an unresolvable login is one whole-batch error naming it rather than N per-repo markers — though a throttle that precedes the author's resolution can pre-empt that error, the login then surfacing on retry; an empty or oversized `repos` list, a malformed or duplicate slug, and an unparseable or inverted window are each rejected before any fetch. Like `authored_activity` it reads no manifest conventions and inherits the operator's `gh` credentials, so it can measure private repositories. It returns per-repo facts only — summing across repositories, ranking, and the attention verdict stay caller-side. (#46)
- `authored_activity` MCP tool: given an explicit `owner`/`repo`, an `author` login, and a window (`since`, with an optional `until` defaulting to now, both RFC3339), returns six decomposed, objective counts of what that user authored and engaged with in the repository — `commitsAuthored`, `issuesOpened`, `pullRequestsOpened`, `reviewsSubmitted` (others' PRs, not the author's own), `pullRequestsEngaged` (commented, not authored), and `issuesEngaged` — for a cross-project attention audit to weigh and render. The counts are never summed; each ships with a per-category `fidelity` label, because precision differs: `commitsAuthored` is the default-branch commit count attributed to the author's linked identity (so it misses squash-merged and email-unlinked commits), while the five search-derived counts are search-index-approximate and — for reviews and engagement — windowed by the item's activity rather than the exact comment/review date. Unlike the other tools it reads no manifest conventions (it is author- and window-driven) and inherits the operator's `gh` credentials, so it can measure private repositories the user-rooted contributions query cannot. It measures a single `owner`/`repo` per call (a caller loops to cover several). An unresolved author login is a named error, never six silent zeros, and a malformed or inverted window is rejected before any fetch. (#45)
- The MCP server now registers a human-readable display title (`Overstory`) alongside its programmatic name (`overstory`), so MCP clients that surface a display name show the capitalized project name while the machine identifier stays lowercase. (#38)
- `backlog_review` now also returns the `criticalPath` block, the same critical-path / gate reduction `project_summary` surfaces: when a repository's manifest declares an ordered critical path, the open critical-path-labeled issues grouped by stream in declared order, each stream with its members and a `gateCleared` signal, plus the off-path/unareaed counts. It reduces the open-issue corpus the grooming read already fetches (no extra fetch) and reports `configured: false` as a no-op when the convention is absent. (#36)
- `project_summary` now also returns a `criticalPath` block: when a repository's manifest declares an ordered critical path, the open critical-path-labeled issues grouped by stream in declared order, each stream carrying its open member issues and a `gateCleared` signal (true when no open critical-path issue remains in it, so a downstream stream may begin). Streams reuse the `areaBalance` taxonomy — a stream `simulation` matches an issue labeled `area/simulation` — so the convention adds only an ordered `criticalPath.streams` list and a `criticalPath.label`, declared together or not at all (declaring one without the other, including an explicit empty `streams: []` with a `label`, is a configuration error). Absent the convention the block reports `configured: false` and is a no-op. The gate is a windowed fact: `gateCleared` is computed over the fetched open-issue window before any list cap, and when `fetchTruncated` is set a cleared gate is provisional, so a caller treats it as authoritative only on a complete window. Critical-path issues that match no declared stream are surfaced, not dropped, split by cause — `unareaedCount` (no area label) versus `offPathCount` (a real area outside the declared path, usually a misconfiguration). Members are ascending by issue number, and the per-stream list cap is surfaced via `listTruncated`. (#35)
- `backlog_review`'s deferred issues now each carry a `bodyRefs` field: the distinct `#N` references parsed from the parked issue's body, ascending, with pull-request references and the issue's own number excluded. These are the issue's stated dependencies, so a client can tell whether a parked issue's blocker has since closed — the "ready to activate" signal a milestone-transition render wants. Parsed from GitHub's rendered plaintext body, and non-nil (serialized as `[]`) when an issue references nothing. (#33)
- `milestone_tracks`'s milestones now each carry a `description` field: the verbatim milestone description, surfaced for theme/prose rendering so a client can show a milestone's stated purpose (which lives only in that prose) alongside the parsed tracks. The parsed `tracks` remain the authoritative member structure — clients render the theme from `description` but do not re-parse it for membership. (#33)
- `backlog_review`'s `staleness` block now reports a `deferredExcludedCount`: how many fetched issues were left out of the staleness universe because they carry a deferred label. With it, `staleCount + freshCount + deferredExcludedCount` equals the fetched count. (#31)
- `milestone_tracks` MCP tool: given an explicit `owner`/`repo`, returns the within-milestone priority structure operators encode in each open milestone's description — a stance distinct from `project_summary`'s milestone *progress*. For each open milestone, the parsed tracks in description order, each with its label, an optional raw status annotation (a bold run-in's parenthetical, e.g. `critical-path`, uninterpreted), and its member issue numbers in order (each with a raw status token: `~~` for a struck/abandoned member, a checkbox marker char, or none). Tracks are recognized by manifest-declared markers under `milestoneTracks`: `headingLevels` (default `[2, 3]`; markdown heading depths that start a track — an explicit empty list disables heading markers), `boldRunIn` (default `true`; treats `**Label** (status):` as a track start — set `false` to disable), `labelStoplist` (default a set of common prose-section labels like `Ikigai`/`Why`/`History`, matched case-insensitively, that are never tracks), and `fetchLimit` (default `100`). Both marker shapes degrade cleanly to zero tracks on the prose/empty descriptions most repositories have, so a milestone with no track structure yields no tracks rather than an error. The parser skips fenced and indented code, excludes pull-request references, and distinguishes a bolded issue number (`**#5**`, a member) from a bold run-in label. The milestone fetch (which now also retrieves each milestone's raw-markdown description) marks the block `available: false` with a `rate_limited`/`fetch_failed` reason on failure rather than failing the call, and the result-set limits — milestones listed, tracks per milestone, members per track — are each surfaced, never silently truncated. The server extracts the structure; tier/cut-line ranking judgment stays caller-side, and references that are dependency mentions rather than members (in inline-prose tracks, link text, or HTML blocks) are an accepted, documented imprecision the caller filters. (#29)
- `project_summary` MCP tool: given an explicit `owner`/`repo`, returns compact structured facts for session orientation — "given what's open now, what should I pick up?" — a stance distinct from `backlog_review`'s grooming. Five blocks: a `milestones` block (each open milestone's authoritative open/closed issue counts plus the fetched open issues belonging to it, each milestone flagged when that member list is a floor relative to its open count); an `areaInventory` block (per functional area, the active-vs-deferred split of open issues, areas identified by the same manifest labels/prefixes `backlog_review` uses); a `hygiene` block (four signals over the open issues — missing-area, unmilestoned-and-aged, stale, deferred-without-context — each a count plus a capped list); an `openPRs` block (each open pull request's branch, draft/ready state, CI-rollup status, and inactivity, plus a stale-PR count); and a `recommendations` block (per-issue inputs — bug-labeled, milestone, age, inactivity — a caller ranks "what next" from, the ranking left caller-side). New manifest conventions live under `summary`: `prStalenessDays` (default `14`), `unmilestonedAgeDays` (default `30`), `prFetchLimit` (default `200`), `milestoneFetchLimit` (default `100`), and `bugLabels` (default `[bug]`, an explicit empty list opting out of bug flagging). The open-issue fetch is primary — its failure fails the call, naming a throttle's retry instant — while the milestone and PR fetches are secondary: each degrades its own block to `available: false` with a `rate_limited`/`fetch_failed` reason rather than failing the whole summary, and the reported `rateLimit` budget is the tightest remaining across the three fetches. Every block carries explicit list/fetch truncation seams, so partial data is never read as complete. (#24)
- `backlog_review` now also returns a `trajectory` reduction: for each manifest-declared lookback window (in days), how many issues were created and how many were closed within the last that-many days, and the net (created minus closed) — the "is the backlog growing or shrinking" signal. Windows are cumulative `last-N` lookbacks (a `last-7` issue also counts in `last-30` and `last-90`), configured under `trajectory.windows` (default `[7, 30, 90]`) with a `trajectory.fetchLimit` (default `500`); both ship a generic default, so the block works on any repo out of the box. Unlike the other blocks it reduces a *second* fetch — issues open and closed by recent update — so a count's `fetchTruncated` flag marks when that fetch's cap was hit before the widest window was covered, making the counts lower bounds. If the second fetch fails, the block degrades to `available: false` with an `unavailable` reason (e.g. `rate_limited`, `fetch_failed`) and the other blocks still return, rather than failing the whole review; a throttle surfaces its recovery time in the `rateLimit` block. Reopened issues read as open (GitHub clears their close), so a close later reopened does not count as a closure — the net-as-of-now reading. The block is aggregate, so the per-call `limit` does not apply to it. (#20)
- `backlog_review` now also returns a `crossRef` reduction: groups of open issues that reference one another issue-to-issue via GitHub's cross-reference graph — candidate consolidation when a topic is scattered across several open issues — each group carrying its member issues (number, title, URL) and the directed reference edges (`from` → `to`) as evidence. References come from GitHub `CrossReferencedEvent`s (pull-request and cross-repository sources excluded), and the connected groups of two or more are reported largest first. The reduction runs on every repository with no configuration. The block reports the total group count, the number of linked issues, and the largest group's size, with explicit list, fetch, and per-issue reference truncation seams. Like overlap, cross-references are computed only over the fetched window, so an edge to an issue outside the window is not detected — the truncation flags distinguish this from complete data. (#19)
- `backlog_review` now also returns an `overlap` reduction: groups of open issues with similar titles — candidate duplicates a maintainer can consolidate — each group carrying its member issues (number, title, URL) and the words its titles share as evidence. Titles are linked by char-trigram Sørensen–Dice similarity over a manifest-declared `overlap.titleSimilarityThreshold` (default `0.5`; `0` disables the reduction, `1` requires an exact match), and the connected groups of two or more are reported largest first. The default works on any repo out of the box — no repo-specific vocabulary needed. The block reports the total group count, the number of overlapping issues, and the largest group's size, with explicit list and fetch truncation seams. Note: overlap is computed only over the fetched window, so a duplicate whose pair sits outside the window is not detected — the truncation flag distinguishes this from the count-based blocks. (#17)
- `backlog_review` now also returns a `rateLimit` block on success: the GitHub GraphQL points budget observed during the fetch — points remaining in the current window and the instant it resets — so a caller can pace itself. The block is omitted when the response carried no budget, so an unknown budget is never mistaken for a present one. (#15)
- `backlog_review` now also returns a quality reduction: open issues that fail one or more grooming checks, reduced to per-check counts plus a granular, most-incomplete-first list (each flagged issue's number, title, URL, age, body length, and which checks it failed) that a caller renders per project. Two checks are universal and on out of the box — a body shorter than `quality.minBodyLength` (default `1`, i.e. must be non-empty; set `0` to disable the body check) and no labels at all. A third, label-coverage by category, is repo-specific and declared under `quality.requiredCategories`: each entry names a label family by an explicit `labels` list and/or `prefixes` rules (e.g. `type` + `/` for `type/bug`), and the reduction reports, per family, how many open issues carry none of its labels. Categories have no generic default, so a repo that declares none still gets the two universal checks. Per-check counts overlap (a zero-label issue counts in both the no-labels and every missing-category total), so they need not sum to the flagged total; list and fetch limits are surfaced explicitly. (#12)
- `backlog_review` now also returns an area-balance reduction: the distribution of open issues across a repository's functional areas, with first-class counts of unclassified (un-area-labeled) and multi-area issues, ordered by count. Areas are identified by manifest conventions under `areaBalance` — an explicit `labels` list and/or `prefixes` rules (each a `prefix` plus a configurable `delimiter`, e.g. `area` + `/` for `area/networking`), matched case-insensitively and unioned, with casing/source variants of one area collapsed into a single bucket. Unlike deferred review, area balance ships generic default prefixes (`area/`, `area:`, `area-`), so a repository with no `areaBalance` block still classifies common `area/*`-style labels out of the box; an explicit `prefixes: []` disables them. Per-area counts may overlap, since a multi-area issue counts in each area. (#10)
- `backlog_review` now also returns a deferred-issue review: open issues carrying a maintainer-declared "deferred" label (e.g. `deferred`, `blocked`, `icebox`) reduced to compact facts — a count, the parked issues (number, title, URL, the labels matched, days since last human activity, age), the configured labels echoed back, and explicit list/fetch truncation seams — ordered most-inactive first. Deferred labels are repo-specific and declared per repository in the manifest under a `deferred.labels` list; there is no generic default, so a repo that declares none reports the block as not-configured rather than guessing. Labels match case-insensitively. (#8)
- `backlog_review` MCP tool: given an explicit `owner`/`repo`, returns compact structured staleness facts — an exact open-issue count, counts by inactivity band, the stalest issues (number, title, URL, days since last human activity), and the threshold applied with its source (a repository's manifest entry deep-merged over generic defaults, or the defaults). Inactivity is measured from the last non-bot comment — ignoring label, assignment, and bot noise — falling back to creation. List and fetch limits are surfaced explicitly and never silently truncated. (#4)
- GitHub access: issues are fetched in-process via the GitHub GraphQL API, authenticated with the operator's existing `gh` credentials (`gh auth token`) — no separate token configuration. GitHub.com only for now. (#4)
- Manifest resolution: per-repo manifests are discovered from `$XDG_CONFIG_HOME/overstory/manifests.d/*.yml` (and `*.yaml`), or from a colon-separated file list in `OVERSTORY_MANIFESTS` when set; entries are keyed by `owner/repo` and deep-merged over generic defaults. A repo's `owner/repo` key must be well-formed and defined exactly once across the discovered files — a malformed key (e.g. containing whitespace) or a duplicate (across files or case-insensitively within one file) is a hard configuration error rather than a silent fallback or drop. (#4, #6)

### Changed

- Staleness now means *neglected*, not merely inactive: an issue carrying a deferred label is quiet by design, so both `backlog_review`'s `staleness` block (its `staleCount`, `freshCount`, `buckets`, and stale-issue list) and `project_summary`'s hygiene `stale` signal now exclude deferred issues rather than counting them as stale. The server makes this distinction once, so a caller no longer re-derives it; the parked-but-cold cue remains visible in the deferred-issue facts. Applies only when a repo configures `deferred.labels`; with no deferred convention, staleness is unchanged. (#31)
- A throttled `backlog_review` now names a recovery time: when GitHub rate-limits the fetch and provides a reset signal (the `Retry-After` / `X-RateLimit-Reset` headers, or the GraphQL `rateLimit` reset on a secondary limit), the tool error names the absolute instant the caller can retry at — so a transient throttle reads as recoverable rather than as a permanent failure. When no signal is available the message degrades to the plain rate-limit error. (#15)
- `backlog_review` output is now a composite of named reduction blocks. Review-level identity (`repo`, `generatedAt`) stays at the top level; the staleness facts move under a `staleness` block, and the new deferred-issue facts arrive under a `deferred` block. Callers that read staleness fields at the top level must now read them under `staleness`. (#8)

### Fixed

- `backlog_review` and `project_summary` no longer fail on a large repository by exceeding the MCP client's tool-result token cap. Each composite reduction now bounds its total response to a byte budget: when the assembled facts would be too large, it trims the detail item-lists — flat lists by item, overlap/cross-reference by whole group, milestone member-lists within each milestone (keeping every milestone's headline progress entry) — largest-contributor-first so trimming is balanced rather than gutting the richest block, while every block's counts, the `openIssueSet`, the critical-path gate signal, and summary fields are preserved. A trimmed response carries a new top-level `sizeBound` marker (`maxBytes`, `finalBytes`, and per-block `{block, dropped, remaining}`); a response that fits is byte-identical to before (no marker). The budget is the new `response.maxBytes` manifest knob (default `20000` per serialization, conservative because the SDK emits the facts twice on the wire; per-repo overridable, rejected below a `4096` floor). When even the irreducible content exceeds the budget the bound is best-effort and `finalBytes` reports the overflow rather than silently claiming success. (#74)
