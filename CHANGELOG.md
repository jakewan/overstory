### Added

- `backlog_review` now also returns a quality reduction: open issues that fail one or more grooming checks, reduced to per-check counts plus a granular, most-incomplete-first list (each flagged issue's number, title, URL, age, body length, and which checks it failed) that a caller renders per project. Two checks are universal and on out of the box â€” a body shorter than `quality.minBodyLength` (default `1`, i.e. must be non-empty; set `0` to disable the body check) and no labels at all. A third, label-coverage by category, is repo-specific and declared under `quality.requiredCategories`: each entry names a label family by an explicit `labels` list and/or `prefixes` rules (e.g. `type` + `/` for `type/bug`), and the reduction reports, per family, how many open issues carry none of its labels. Categories have no generic default, so a repo that declares none still gets the two universal checks. Per-check counts overlap (a zero-label issue counts in both the no-labels and every missing-category total), so they need not sum to the flagged total; list and fetch limits are surfaced explicitly. (#12)
- `backlog_review` now also returns an area-balance reduction: the distribution of open issues across a repository's functional areas, with first-class counts of unclassified (un-area-labeled) and multi-area issues, ordered by count. Areas are identified by manifest conventions under `areaBalance` â€” an explicit `labels` list and/or `prefixes` rules (each a `prefix` plus a configurable `delimiter`, e.g. `area` + `/` for `area/networking`), matched case-insensitively and unioned, with casing/source variants of one area collapsed into a single bucket. Unlike deferred review, area balance ships generic default prefixes (`area/`, `area:`, `area-`), so a repository with no `areaBalance` block still classifies common `area/*`-style labels out of the box; an explicit `prefixes: []` disables them. Per-area counts may overlap, since a multi-area issue counts in each area. (#10)
- `backlog_review` now also returns a deferred-issue review: open issues carrying a maintainer-declared "deferred" label (e.g. `deferred`, `blocked`, `icebox`) reduced to compact facts â€” a count, the parked issues (number, title, URL, the labels matched, days since last human activity, age), the configured labels echoed back, and explicit list/fetch truncation seams â€” ordered most-inactive first. Deferred labels are repo-specific and declared per repository in the manifest under a `deferred.labels` list; there is no generic default, so a repo that declares none reports the block as not-configured rather than guessing. Labels match case-insensitively. (#8)
- `backlog_review` MCP tool: given an explicit `owner`/`repo`, returns compact structured staleness facts â€” an exact open-issue count, counts by inactivity band, the stalest issues (number, title, URL, days since last human activity), and the threshold applied with its source (a repository's manifest entry deep-merged over generic defaults, or the defaults). Inactivity is measured from the last non-bot comment â€” ignoring label, assignment, and bot noise â€” falling back to creation. List and fetch limits are surfaced explicitly and never silently truncated. (#4)
- GitHub access: issues are fetched in-process via the GitHub GraphQL API, authenticated with the operator's existing `gh` credentials (`gh auth token`) â€” no separate token configuration. GitHub.com only for now. (#4)
- Manifest resolution: per-repo manifests are discovered from `$XDG_CONFIG_HOME/overstory/manifests.d/*.yml` (and `*.yaml`), or from a colon-separated file list in `OVERSTORY_MANIFESTS` when set; entries are keyed by `owner/repo` and deep-merged over generic defaults. A repo's `owner/repo` key must be well-formed and defined exactly once across the discovered files â€” a malformed key (e.g. containing whitespace) or a duplicate (across files or case-insensitively within one file) is a hard configuration error rather than a silent fallback or drop. (#4, #6)

### Changed

- `backlog_review` output is now a composite of named reduction blocks. Review-level identity (`repo`, `generatedAt`) stays at the top level; the staleness facts move under a `staleness` block, and the new deferred-issue facts arrive under a `deferred` block. Callers that read staleness fields at the top level must now read them under `staleness`. (#8)
